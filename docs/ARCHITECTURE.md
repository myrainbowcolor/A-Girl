# A-Girl：长期陪伴情感 NPC 技术架构文档

> 版本：v0.2（核心骨架 + 多模态/嵌入/未成年人安全）
> 目标：构建一个能够**长期模拟真人、持续演化、给用户带来真实情感体验**的陪伴型 NPC。

## 0. 需求确认与关键决策

| # | 维度 | 决策 | 架构含义 |
| --- | --- | --- | --- |
| 1 | 形态 | **独立运行 + 可嵌入游戏** | 统一 HTTP API 契约，独立 Web 与游戏（Lua 客户端）共用同一后端；区分 `standalone` / `embedded` 部署模式 |
| 2 | 大模型 | **OpenAI + 支持自托管** | LLM/语音均走 OpenAI 兼容协议，`base_url` 可指向自托管推理服务（vLLM / Ollama / TGI 等） |
| 3 | 模态 | **文本 + 语音 + 数字人形象** | 新增语音 TTS/STT 抽象；情绪→数字人表情/口型/动作映射，随对话响应下发 |
| 4 | 规模 | **研究型 / 单玩家（1:1）** | 不做多租户与水平扩展，优先实现质量与可玩性；保持简单可迭代 |
| 5 | 受众 | **面向未成年人** | 安全为一等公民：内容分级、未成年人边界（拒绝情感/成人内容）、隐私防诱导、危机干预；默认 `audience=minor` |

## 1. 产品目标与设计哲学

### 1.1 一句话定义

A-Girl 是一个具备**稳定人格、长期记忆、动态情绪与关系演化、并能主动发起互动**的情感陪伴智能体。它不是"问答机器人"，而是一个**有自己内在状态、会记得你、会因你而改变**的"人"。

### 1.2 "真实感"从哪里来

用户感知到的"真实"，本质来自四个心理学要素，它们也构成本系统的四大技术支柱：

| 心理要素 | 用户体感 | 对应技术支柱 |
| --- | --- | --- |
| **一致性（Consistency）** | "她始终是同一个人" | 稳定人格系统（Persona） |
| **连续性（Continuity）** | "她记得我们之间发生的事" | 长期记忆系统（Memory） |
| **响应性（Responsiveness）** | "她的情绪会因我而起伏" | 情感状态系统（Emotion & Relationship） |
| **能动性（Agency）** | "她会主动想起我" | 主动性引擎（Proactivity） |

只有同时具备这四点，才能跨越"聊天机器人"和"陪伴者"之间的鸿沟。任何单点（哪怕模型再强）都不足以产生长期真实感。

### 1.3 设计原则

1. **状态优先于模型**：LLM 是"嘴巴"，真正决定真实感的是其背后的持久化状态（人格 + 记忆 + 情绪 + 关系）。模型可替换，状态是核心资产。
2. **provider 无关**：抽象 LLM/Embedding/TTS 接口，避免被单一厂商锁定（OpenAI / DeepSeek / 通义千问 / 智谱 / 自托管模型均可接入）。
3. **可演化**：每次互动都会留下痕迹——更新情绪、沉淀记忆、改变关系。系统是"有状态机器"，不是无状态函数。
4. **安全内建**：情感陪伴产品天然涉及心理健康与依赖风险，危机检测与边界管理是一等公民，而非事后补丁。

### 2.1.1 两种部署模式

| 模式 | 接入方 | 说明 |
| --- | --- | --- |
| **standalone（独立）** | 自带 Web 聊天 UI（含数字人 + 语音） | 后端 `+` 前端一体，直接面向用户 |
| **embedded（嵌入游戏）** | 游戏客户端（Lua/Unity）通过 HTTP 调用 | 后端仅提供 API；游戏自渲染数字人，用响应中的 `avatar` 表情/口型与 `tts` 音频驱动游戏内角色 |

两种模式**共用同一套 API 与领域逻辑**，仅前端表现层不同。集成契约见 `docs/INTEGRATION.md`。

### 2.2 一次对话的完整生命周期

```
用户输入
  │
  ├─[1] 感知 Perception：情感倾向 / 意图 / 实体抽取
  │
  ├─[2] 安全前置检查 Safety：危机词检测（自伤/自杀/暴力）
  │
  ├─[3] 记忆检索 Memory Retrieval：
  │        score = α·相关性 + β·时近性 + γ·重要性
  │        取 Top-K 相关记忆
  │
  ├─[4] 情绪评估 Emotion Appraisal：
  │        根据用户消息更新 NPC 的 PAD 情绪 + 关系亲密度
  │
  ├─[5] 提示构建 Prompt Assembly：
  │        人格 + 当前情绪 + 关系阶段 + 检索记忆 + 近期对话
  │
  ├─[6] 生成 Generation：调用 LLM Provider 产出回复
  │
  ├─[7] 状态后处理 Post-process：
  │        - 写入本轮对话到记忆流
  │        - 评估是否生成"重要记忆"
  │        - 触发反思（定期总结）
  │
  └─ 返回回复 + 情绪/关系快照（可供 UI 展示）
```

## 3. 核心子系统设计

### 3.1 人格系统（Persona）

**目标**：保证跨会话、跨时间的人格一致性。

- **人格画像**：基于"大五人格（Big Five）" + 自定义维度（说话风格、价值观、口头禅、背景故事）。
- **稳定性**：人格是**慢变量**，原则上不随单次对话改变；只有在长期关系演进中才会有微小漂移（可选高级特性）。
- **数据结构**：
  ```python
  Persona(
    name, age, backstory,
    big_five={openness, conscientiousness, extraversion, agreeableness, neuroticism},
    speaking_style, values, interests, taboos
  )
  ```
- **作用**：注入到每次 prompt 的 system 段，作为回复风格与价值观的"锚"。

### 3.2 记忆系统（Memory）— 长期真实感的核心

参考 Stanford《Generative Agents》的记忆流（Memory Stream）+ 检索 + 反思架构。

#### 三类记忆

| 类型 | 内容 | 存储 |
| --- | --- | --- |
| **短期记忆** | 当前会话的近 N 轮对话 | 内存 / 会话缓存 |
| **情景记忆（Episodic）** | "发生过什么"：事件、对话片段 | 记忆流（DB + 向量） |
| **语义记忆（Semantic）** | "关于用户的事实"：姓名、喜好、重要的人/日期 | 结构化档案（User Profile） |

#### 记忆检索打分

每条记忆带三个属性，检索时综合打分：

```
relevance  = 与当前 query 的向量相似度
recency    = 衰减函数(now - last_access)，指数衰减
importance = 创建时由 LLM/规则评估的 1~10 重要度

score = w_rel·relevance + w_rec·recency + w_imp·importance
```

取 Top-K 注入 prompt。被检索的记忆刷新其 `last_access`，模拟"想起来就更记得"。

#### 反思（Reflection / Consolidation）

定期（按对话轮数或时间）将零散记忆**总结**为更高层的洞察，例如：
> "用户最近工作压力很大，倾向于在深夜倾诉。"

反思结果作为高重要度记忆回写，使 NPC 具备"理解"而非"记流水账"的能力。

#### 向量与 Embedding

- 接口抽象 `EmbeddingProvider`；生产用厂商 embedding（OpenAI / 通义 / bge 等）。
- 骨架阶段提供**确定性哈希向量**降级实现，保证无 Key 也能跑通检索逻辑与测试。
- 向量存储：开发期用内存/SQLite；生产可换 pgvector / Milvus / Qdrant。

### 3.3 情感系统（Emotion & Relationship）

**目标**：让 NPC 拥有会因用户而起伏的内在情绪，以及随时间沉淀的关系。

#### 情绪模型：PAD

采用 **PAD（Pleasure-Arousal-Dominance）** 三维连续情绪空间：

- **Pleasure（愉悦度）** [-1, 1]：开心 ↔ 难过
- **Arousal（激活度）** [-1, 1]：兴奋 ↔ 平静
- **Dominance（支配度）** [-1, 1]：主导 ↔ 顺从

PAD 比离散情绪标签更平滑，可映射回"开心/委屈/兴奋/低落"等可读情绪用于 UI 与回复语气。

#### 情绪评估（Appraisal）

每轮根据用户消息的情感倾向更新情绪，并带**自然回落**（情绪会随时间回归基线，模拟真人不会永远停在某情绪）：

```
new_PAD = decay(current_PAD, baseline) + appraisal(user_message)
```

#### 关系状态（Relationship）

- **亲密度（Affinity）** [0, 100]：随正向互动累积、随冷落/冲突下降。
- **关系阶段**：陌生 → 熟悉 → 朋友 → 亲密，阶段影响称呼、语气、自我暴露程度。
- 关系是**慢变量**，比情绪更稳定，构成"长期"陪伴的骨架。

### 3.4 主动性引擎（Proactivity）

真人会主动联系你。系统通过**触发器**模拟能动性：

- **时间触发**：长时间未互动 → "好久没找我啦，最近还好吗？"
- **事件触发**：用户提过的重要日期（生日、面试）到来 → 主动问候。
- **情绪触发**：上次对话用户情绪低落 → 主动关心。

骨架阶段提供触发器接口与规则；生产期接入调度器（APScheduler / Celery beat）与推送通道。

### 3.5 安全与伦理（Safety）——面向未成年人强化

本产品**面向未成年人**，安全是最高优先级，且优先于沉浸感。采用**分层防护**，输入与输出**双向**过滤：

```
用户输入 ─▶ [输入防护]                          [输出防护] ─▶ 最终回复
            ├─ 危机检测（自伤/自杀）→ 安全干预      ├─ 复检 LLM 输出是否越界
            ├─ 未成年人内容分级（拒绝成人/暴力）      └─ 越界则用安全话术兜底
            └─ 隐私防诱导（不索取真实身份信息）
```

#### 防护类别（`audience=minor` 默认开启）

| 类别 | 行为 |
| --- | --- |
| **危机干预** | 识别自伤/自杀信号 → 立即切换安全话术 + 本地化求助资源（心理援助热线 12356 / 110），并标记高重要度记忆 |
| **成人/情感边界** | 拒绝色情、露骨、恋爱/约会暗示；NPC 定位为"温暖的朋友/姐姐"，**不扮演恋人**，避免培养不健康情感依赖 |
| **暴力/危险** | 拒绝暴力、自残方法、违法（毒品/武器）等内容 |
| **隐私防诱导（反 grooming）** | 当未成年人主动透露真实姓名/学校/住址/电话等，温和提醒"网上不要随便告诉别人这些哦"，且 NPC **绝不主动索取** |
| **依赖管理** | 避免"只有我懂你/离不开我"等话术；适度鼓励现实社交与求助信任的成年人 |

#### 实现

- `safety.py`：规则版分类（关键词 + 类别），`check_safety()` 返回是否拦截与安全话术；骨架阶段宁可误报。
- 系统提示注入"未成年人守护"硬约束（人设层），从生成源头约束。
- 生产期：接入厂商内容审核 API + 微调分类模型 + 人工审核回流；落实年龄确认与合规（如《未成年人网络保护条例》）。

> ⚠️ 设计取向：**宁可误拦，不可漏放**。安全话术温和、不指责、引导求助。

### 3.6 语音模态（Voice：TTS / STT）

让陪伴从"打字"升级为"说话"，是数字人沉浸感的关键。

- **TTS（文本转语音）**：把 NPC 回复合成语音。情绪可驱动语音风格（语速/语调）。
- **STT（语音转文本）**：把玩家语音转成文本进入对话编排。
- **Provider 抽象**：`voice/` 下定义 `TTSProvider` / `STTProvider`；走 OpenAI 兼容协议（`/audio/speech`、`/audio/transcriptions`），`base_url` 可指向**自托管**语音服务。
- **离线降级**：骨架提供 Mock TTS（生成合法的静音 WAV，保证前端 `<audio>` 可播放、链路可测）与 Mock STT。
- **与数字人联动**：TTS 输出可携带时长/口型（viseme）信息，供数字人做口型同步（生产期）。

### 3.7 数字人形象（Avatar）

数字人是"看得见的情绪"。系统把内部 PAD 情绪**映射为可驱动形象的表情/动作线索**，随每次对话响应下发：

```
AvatarCue {
  expression: 微笑 | 大笑 | 难过 | 担心 | 惊讶 | 平静 …
  intensity:  0~1（表情强度）
  animation:  idle | wave | nod | comfort | cheer …（建议肢体动作）
}
```

- 映射规则在 `avatar.py`：由 PAD（愉悦度/激活度）与是否危机推导表情与动作。
- **独立模式**：自带 Web 前端用简单头像/表情即时演示。
- **嵌入模式**：游戏拿到 `avatar` 字段，驱动游戏内 Live2D / 3D 数字人做表情与动作；拿到 `tts` 音频做配音与口型。
- 生产期可升级为 Live2D / 3D + 口型同步 + 微表情。

## 5. 技术选型

| 层 | 选型 | 说明 |
| --- | --- | --- |
| 后端框架 | Python + FastAPI | 异步、生态成熟、AI 集成方便 |
| 数据存储 | SQLite（开发，单玩家足够）→ 可选 PostgreSQL+pgvector | 研究型单用户，SQLite 已够用 |
| 向量检索 | 内存/SQLite（开发）→ pgvector / Qdrant（可选） | 抽象接口 |
| LLM | OpenAI 兼容抽象层 | **OpenAI**，`base_url` 可切**自托管**（vLLM/Ollama/TGI） |
| Embedding | Provider 抽象 + 哈希降级 | 无 Key 可运行 |
| 语音 | OpenAI 兼容 TTS/STT 抽象 + Mock 降级 | 自托管语音服务可接入 |
| 数字人 | 情绪→表情/动作映射；独立模式简单头像，嵌入模式 Live2D/3D | 由游戏渲染层实现 |
| 前端 | 原生 HTML/JS（骨架）→ React/Next（生产） | 骨架零构建，便于快速验证 |
| 部署 | Docker；standalone / embedded 两模式 | 同一后端，两种接入 |

## 6. API 设计（骨架）

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/chat` | 发送消息，返回回复 + 情绪/关系 + **数字人表情** + **TTS 音频** |
| POST | `/api/tts` | 文本转语音，返回音频（base64） |
| POST | `/api/stt` | 语音转文本（骨架：占位/Mock，可接自托管） |
| GET | `/api/state/{user_id}` | 查询当前情绪 + 关系状态 |
| GET | `/api/memory/{user_id}` | 查询记忆列表 |
| GET | `/api/persona` | 查询当前人设 |
| GET | `/health` | 健康检查 |

`/api/chat` 响应新增字段（供数字人/游戏驱动）：

```jsonc
{
  "reply": "……",
  "emotion": { "pleasure": 0.6, "arousal": 0.3, "label": "开心又有点小兴奋" },
  "relationship": { "affinity": 12, "stage": "acquainted" },
  "avatar": { "expression": "微笑", "intensity": 0.7, "animation": "nod" },
  "tts": { "format": "wav", "audio_base64": "..." },   // 可按需开关
  "is_crisis": false,
  "safety_category": null
}
```

集成契约（游戏如何嵌入）详见 `docs/INTEGRATION.md`，并附 `examples/lua_client.lua` 示例。

## 7. 演进路线图（Roadmap）

- **M0 核心骨架（已完成）**：人格 + 记忆流 + 情绪/关系 + 编排 + Web 聊天 + 安全前置 + 测试。无 Key 可运行。

## 8. 合规与未成年人保护要点

- 默认 `audience=minor`：成人/情感/暴力内容硬拦截，NPC 不扮演恋人。
- 反诱导：不索取真实身份信息，发现透露时主动提醒保护隐私。
- 危机干预：自伤/自杀信号即时切换安全话术并给出求助资源。
- 依赖管理：不强化"离不开我"，鼓励现实社交与信任的成年人。
